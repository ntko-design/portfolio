Dropdown Component

Internal Design System Component | April 2023

A design system project initially focused on responsive dropdown sizes. It grew into a comprehensive initiative that revamped the design component and backend code to standardize menu properties, simplify implementation, and enhance the end user experience.

Domain Problem

Klue's design system has evolved constantly due to frequent updates, resulting in increased complexity and redundancy with variants. In response, volunteers in the company formed the Design System Guild, which held bi-monthly meetings to discuss and fix design system issues.

As a Guild member, I examined existing dropdowns in Klue to inform the design of responsive mockups. I noticed sizing and property inconsistencies across each dropdown menu instance, seen in areas like font sizes, drop shadows, and dividers between sections.

Precedent Analysis

For my first pass, I documented each dropdown property to identify patterns and grouped them into three sizes. Seeing an opportunity to streamline design implementation, I created a responsive component that included all of Klue's unique dropdown elements.

Design Review

I then lead a review session to collect feedback from the design team. I demonstrated the versatility of my design component by showcasing how it could replicate any existing dropdown menu in the Klue app—albeit with normalized properties—by enabling various options.

Engineering Review

Taking this to the Design Systems Guild for another review session, we discussed how my approach could be taken further to help speed up developer implementation. As a team, we decided to create a separate component for developer specification purposes.

Solution

Design Component

The design team has access to a streamlined dropdown component with two sizing and three common selection element variables. It also includes additional boolean properties to enable elements like headings, close buttons, and tabs to fit the context of the interface.

Documentation

This allows designers to quickly produce dropdowns that align with design system standards, all while eliminating the need for extensive documentation for each instance they add.

Developer Component

The developer component is divided into a header, footer, and content area, with the former two having options to toggle on or off based on interface requirements. The content area differs as it can accommodate virtually any interface, but it also offers prebuilt options for quicker implementation of common elements to reduce the need for complex customization.

Value

Why is it needed?

The consolidation of well over 10 dropdown variants into a single component accelerated both design and development processes by reducing the time needed to design, code, and document. The normalized visuals and interactions helps deliver a unified end user experience, while the addition of size variants fulfilled the project's responsive design goals.

Final Thoughts

The project's evolution beyond its initial requirements resulted in a highly beneficial solution for the product team. This experience taught me the importance of being proactive and how considering aspects beyond project briefs can improve project outcomes. It also provided insights on effective collaboration with developers as it encouraged me to deeply understand their work process.

If I had more time to delve further into this project, I would have liked to focus more on the developer side of it. I believe that there may be opportunities to make the content area even more efficient by enabling developers to quickly generate predetermined layouts using the most common components. This could potentially provide even greater benefits for my colleagues.
